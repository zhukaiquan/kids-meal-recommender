# Kids Meal Recommender 中文化与图片化设计

日期：2026-03-27

## 背景

当前版本已经具备基础的三餐推荐、食物库管理和历史记录能力，但仍然偏向英文工具界面，视觉也更接近管理台，而不是家长日常会反复打开使用的亲子小工具。

这一轮迭代的目标不是增加新的排餐逻辑，而是把现有产品升级成更适合中文用户、视觉更友好、并且更直观的“看图选餐”体验。

## 目标

- 全站界面改为中文
- UI 从工具风升级为儿童友好、插画感更强的视觉风格
- 食物条目支持封面图
- 新增和编辑食物时支持联网搜索图片并选择封面
- `今日推荐`、`食物库`、`历史记录` 都展示食物图片

## 非目标

这一轮不做以下内容：

- 不增加新的排餐规则或营养评分逻辑
- 不做多孩子支持
- 不做账号系统和云端同步
- 不做图片本地上传持久化
- 不做后端服务
- 不做 AI 自动识图或图片生成

## 产品范围

### 1. 今日推荐

`今日推荐` 页面保留早餐、午餐、晚餐三餐结构，但视觉改成更偏绘本式的大卡片布局。

每张餐卡需要包含：

- 中文餐名
- 1 句轻量说明文案
- 当前推荐中的食物图片缩略图
- 当前推荐中的食物中文名称
- 明显的“换一个”按钮

交互规则保持现有逻辑：

- 默认每天生成一次三餐
- 单餐刷新时，只刷新当前餐
- 其它两餐保持不变

### 2. 食物库

食物库从偏表格的管理形式，升级为图片卡片式列表。

每张食物卡需要展示：

- 封面图
- 中文名称
- 适用餐型
- 标签
- 编辑按钮
- 删除按钮

食物没有图片时，显示统一的插画风占位图形，不出现空白框。

### 3. 新增与编辑食物

新增和编辑食物时，表单分为两部分：

#### 基础信息

- 食物名称
- 适用餐型
- 标签
- 是否启用

#### 图片选择

默认流程：

1. 输入食物名称
2. 点击“搜索图片”
3. 系统使用食物名组合“插画 / 卡通 / 食物”等关键词发起公开图片搜索
4. 展示 6 张候选图
5. 用户点选其中 1 张作为封面

兜底规则：

- 如果没有满意结果，允许“不设置图片”直接保存
- 图片搜索结果只用于选择封面，不在本地保存原始二进制文件

### 4. 历史记录

历史记录继续按天展示，但每一天改成更易浏览的图片卡片：

- 日期
- 早餐图片与名称
- 午餐图片与名称
- 晚餐图片与名称

目标是让历史页从“日志”变成“可回看的每日菜单”。

## 图片来源方案

这一轮采用公开开放版权图片来源，优先使用 `Wikimedia Commons` 公开 API 进行图片搜索与展示。

选择理由：

- 官方开放 API，可直接查询图片与元数据
- 公开可访问，不引入自建后端
- 可返回来源页、作者、许可等信息，后续可扩展来源说明

参考文档：

- [Commons:API/MediaWiki](https://commons.wikimedia.org/wiki/Commons:API/MediaWiki)
- [Help:Machine-readable data](https://commons.wikimedia.org/wiki/Help:Machine-readable_data)
- [API:Web APIs hub](https://www.mediawiki.org/wiki/API:Web_APIs_hub/API_index)

搜索策略：

- 主查询词：`食物名 + 插画 + 卡通 + 食物`
- 如果中文结果较弱，自动追加英文备选查询
- 每次返回最多 6 个候选图
- 前端只保存被选中的图片元信息，不缓存完整图库结果

## 数据结构调整

现有 `FoodItem` 在文字字段之外，新增图片元信息字段：

```ts
type FoodImage = {
  thumbnailUrl: string;
  fullUrl: string;
  sourceName: "wikimedia-commons";
  sourcePageUrl: string;
  authorName: string | null;
  license: string | null;
  searchQuery: string;
};

type FoodItem = {
  id: string;
  name: string;
  mealTypes: MealType[];
  tags: FoodTag[];
  enabled: boolean;
  image: FoodImage | null;
};
```

兼容性要求：

- 老数据没有 `image` 字段时，读取后按 `null` 处理
- 历史记录继续依赖名称快照，不额外回填旧历史图片

## 视觉设计方向

整体风格定为“亲子绘本感”，避免后台工具气质。

视觉要求：

- 主色调使用奶油白、浅橙、淡黄、草绿
- 使用大圆角卡片和柔和阴影
- 重要信息优先用卡片、标签和图片组织，不用大段纯文字
- 标签做成彩色胶囊
- 页面背景允许使用轻量装饰图形，例如云朵、星星、圆点或波浪分隔

视觉边界：

- 不做过度幼稚的拟物玩具风
- 不使用过深色大面积背景
- 不使用成人效率工具风的灰白表格布局

## 技术实现约束

- 保持 `React + TypeScript + Vite`
- 继续使用浏览器本地存储
- 图片搜索通过前端直接请求公开 API
- 图片搜索逻辑与推荐逻辑解耦，单独放在服务模块
- 页面中文文案集中整理，避免散落硬编码

## MVP 验收标准

- 打开站点后，主界面为中文
- `今日推荐` 页面三餐卡片能展示图片或占位图
- 食物库列表能展示图片卡片
- 新增食物时可以搜索图片并选择封面
- 搜不到图时仍可保存食物
- 历史记录页面能展示带图片的每日餐食记录
- 原有推荐、刷新、历史、存储逻辑不被破坏

## 风险与取舍

### 1. 搜图质量波动

公开图片搜索结果质量会受关键词影响，某些食物可能搜不到理想的卡通图。

取舍：

- 第一版接受人工挑图
- 允许无图保存

### 2. 第三方图片外链稳定性

图片来自第三方公开来源，未来可能出现部分链接失效或加载慢。

取舍：

- 第一版先接受外链图片
- 后续如果频繁出现问题，再考虑本地缓存或代理层

### 3. 图片上传复杂度

如果本轮加入图片本地上传并长期保存，会把存储复杂度从 `localStorage` 推向 `IndexedDB`，不符合当前 MVP 收口目标。

取舍：

- 本轮不做本地上传持久化
- 优先把中文化、图片搜索和 UI 升级跑通
